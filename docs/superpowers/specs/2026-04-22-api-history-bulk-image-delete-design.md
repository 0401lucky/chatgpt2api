# API 历史页批量删除图片设计

## 背景

当前 API 历史页只能查看、下载和查看提示词，无法手动清理历史图片。随着记录积累，用户需要一种可控的清理方式，能够从当前页中跨记录选择多张图片并批量删除。

现有实现特点：

- 前端历史页按“记录卡片”展示，一条记录下可能有多张图片
- 后端 `ImageHistoryService` 只支持保存、读取和裁剪旧记录，不支持删除指定图片
- 图片文件保存在磁盘目录中，记录索引保存在 `image_history.json`

## 目标

- 在 API 历史页增加“批量管理”模式
- 用户只能在当前页中选择图片
- 支持跨记录批量删除当前页中的多张图片
- 删除成功后同步更新：
  - 图片缩略图
  - 记录的 `image_count`
  - 顶部统计信息
  - 分页结果
- 某条记录删空后，自动从列表中移除

## 非目标

- 不做跨页选择与删除
- 不做“按整条记录删除”
- 不改动浏览器本地图像会话记录
- 不引入新的前端测试框架
- 不处理与本功能无关的历史页现有 lint warning

## 用户交互设计

### 模式切换

历史页保留现有普通浏览模式，并新增批量管理模式。

普通模式下：

- 可查看大图
- 可下载图片
- 可查看完整提示词

批量管理模式下：

- 页面顶部出现操作条
- 当前页所有图片出现可点击的选中状态
- 点击图片切换选中，而不是打开大图
- 下载按钮隐藏或禁用
- 提示词查看入口保留

### 批量管理操作条

进入批量管理模式后，顶部操作条包含：

- 已选图片数量
- `全选当前页`
- `清空选择`
- `删除所选`
- `取消管理`

### 删除确认

点击 `删除所选` 后弹出确认框，明确提示：

- 将永久删除图片文件
- 将同步删除历史记录中的图片引用
- 如果某条记录删空，会自动删除整条记录

## 前端状态设计

历史页新增以下状态：

- `isManageMode`
- `selectedImageKeys`
  - key 格式：`recordId:imageId`
- `isDeleting`
- `deleteConfirmOpen`

前端不做复杂的局部增量状态拼接。删除成功后，直接使用后端返回的最新 `items` 覆盖 `records`，并清理已失效的图片 URL 缓存，避免：

- 已删图片仍显示旧缩略图
- 某条记录删空后残留空卡片
- 分页和统计信息不同步

## 后端接口设计

新增删除接口：

- `POST /api/image-history/delete`

请求体：

```json
{
  "items": [
    {
      "record_id": "record-1",
      "image_ids": ["image-1", "image-2"]
    },
    {
      "record_id": "record-2",
      "image_ids": ["image-3"]
    }
  ]
}
```

这样设计的原因：

- 图片删除天然属于“记录内删除若干图片”
- 使用 `record_id + image_ids` 比平铺图片 id 更稳
- 服务层更容易定位和更新对应记录

响应体建议包含：

- `deleted_images`
- `deleted_records`
- `items`

其中 `items` 为最新历史记录列表，供前端直接覆盖状态。

## 服务层设计

`ImageHistoryService` 新增批量删除方法，职责为：

1. 根据 `record_id` 查找记录
2. 删除记录中命中的图片元数据
3. 删除对应磁盘图片文件
4. 重新计算 `image_count`
5. 如果记录已空，则整条记录移除
6. 一次性保存最新 `image_history.json`

该过程必须在现有 `Lock` 保护下执行，尽量保持元数据和文件删除的一致性。

## 删除语义

接口采用“尽力删除”语义：

- 某些图片不存在：跳过
- 某些记录不存在：跳过
- 只要至少成功删除 1 张图片，就返回成功
- 若本次请求没有删除任何图片，则返回 `404` 或 `400`

这样可以减少批量操作时因局部异常导致整批失败。

## 测试设计

### 后端服务测试

为 `ImageHistoryService` 增加测试：

- 删除单张图片后，记录保留，`image_count` 正确减少
- 删除最后一张图片后，记录被移除
- 对应文件被真实删除

### API 测试

为历史接口增加测试：

- 未授权访问删除接口被拒绝
- 删除多条记录中的多张图片成功
- 删除后返回最新 `items`
- 删除最后一张图片后记录消失
- 删除不存在图片时按“尽力删除”语义处理

## 风险与控制

### 风险 1：缓存 URL 残留

删除图片后，如果前端仍持有旧的 `objectURL`，可能出现已删图片继续显示的情况。

控制方式：

- 删除成功后按最新 `items` 重建当前页缓存
- 清理失效 `recordId:imageId` 对应的 URL

### 风险 2：记录删空后的 UI 残留

控制方式：

- 以后端返回的最新 `items` 为准整体覆盖状态

### 风险 3：批量删除时局部失败影响体验

控制方式：

- 后端采用“尽力删除”语义
- 前端删除成功提示中展示删除数量，而不是要求严格全成全败
